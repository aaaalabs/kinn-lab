Zwei Redis-Clients: @upstash/redis (Großteil des Codes: new Redis({ url, token })) und @vercel/kv (Legacy in Radar-Files: createClient({ url, token })). Beide nutzen dieselben KINNST_KV_REST_API_* Umgebungsvariablen. Niemals Redis.fromEnv() verwenden.
Redis Helper Pflicht: Immer den Singleton aus /api/utils/redis.js verwenden. Kein direktes Initialisieren von Redis-Clients in einzelnen API-Dateien.

Auth & Sicherheit

Authentifizierung & Sicherheitsmodell

Kein Session-Store, kein Rollen-System — Admin oder nicht

Token-Typen

TokenTypPayloadAblaufVerwendung
AdminPlaintextADMIN_PASSWORD—Timing-safe Vergleich via crypto.timingSafeEqual. Kein JWT, kein Hash.
User-JWTJWT (signed){ email, type: 'profile' }NieLong-lived Token für User-Profil-Verwaltung
Confirm-JWTJWT (signed){ email, type: 'confirm' }24 StundenE-Mail-Bestätigung (Double Opt-In)
PrintPlaintextTEAM_PASSWORD—Zugang zum Print/Design-Toolkit (von Admin getrennt)
Radar-CronSecretRADAR_CRON_SECRET—Authentifiziert Vercel Cron-Requests

Security-Header (aus vercel.json)

HeaderWert
X-Frame-OptionsDENY
X-Content-Type-Optionsnosniff
X-XSS-Protection1; mode=block
Referrer-Policystrict-origin-when-cross-origin
Permissions-Policycamera=(), geolocation=() (global) | microphone=(self) (nur /raus/*)
Content-Security-Policydefault-src 'self'; script-src 'self' 'unsafe-inline' fonts.googleapis.com embed.lu.ma; connect-src 'self' *.upstash.io api.lu.ma; frame-src lu.ma embed.lu.ma; frame-ancestors 'none'
Cache-Controlno-cache, must-revalidate (alle .html + Root)
Kein Rollen-System: Es gibt nur zwei Zustände — authentifizierter Admin (via Bearer Token) oder kein Admin. Kein Middleware-Layer, jede Admin-Route prüft selbst. JWTs werden nicht rotiert und nicht widerrufen — kein Token-Store vorhanden.

08
Onboarding

Erste Schritte für neue Entwickler

Von Zero zum ersten PR in fünf Schritten
Schritt 1
Repo clonen & Umgebung einrichten
Repo clonen, npm install ausführen.
.env.local aus .env.example erstellen und mit Werten befüllen.
Alle nötigen API-Keys von Thomas oder einem Core-Team-Mitglied anfordern.
Schritt 2
Lokal starten mit vercel dev
vercel dev startet den lokalen Dev-Server auf Port 3000.
Serverless Functions werden automatisch emuliert — kein separater API-Server nötig.
Kein Build-Schritt, kein Webpack, keine Bundler-Konfiguration.
Schritt 3
Diese Architekturdokumentation lesen
kinn-architecture.html gibt den vollständigen Überblick:
API-Endpoints, Redis-Schema, Token-Typen, Env-Vars, System-Architektur.
Danach vercel.json für Routing- und Infra-Details.
Schritt 4 & 5
Platform TechTalk als KickStart
Bei gemeinsamer Entwicklung wird ein dedizierter TechTalk organisiert — Walk-through durch Architektur, Live-Systeme und offene Baustellen.
Bei Interesse direkt an Thomas wenden: thomas@kinn.at

Wichtige Conventions auf einen Blick

ThemaConvention
ModuleES Modules überall — import/export, kein require()
RedisImmer api/utils/redis.js Singleton, KINNST_KV_ Prefix für Env-Vars
UmlauteEchte Umlaute (ü, ö, ä, ß) im Frontend — kein ASCII-Ersatz
AuthAdmin: Bearer Token; User: JWT in URL-Param oder Header
DesignKein Framework — Vanilla CSS, KINN Brand Design System (Mintgrün, Ecru, Futura)
PIIPublic Repo — kein echtes User-Daten in Code oder Commits
Deploymentvercel --prod für Production — nur nach Review
Kein Build-Schritt nötig: vercel dev kümmert sich um Serverless-Emulation lokal. Was lokal funktioniert, funktioniert in Production — es gibt keine Build-Artefakte oder Transpiling-Schritte.
