04
Risiko-Bewertung

Was kann schiefgehen?

Big-Bang Migration: 100 Dateien in einem PR

Hoch
Big-Bang Deployment
100 Dateien in einem Merge. Wenn ein Endpoint bricht, ist Rollback nur komplett möglich. Kein schrittweises Deployment. Vercel baut alles gleichzeitig — ein TypeScript-Fehler im Build stoppt das gesamte Deployment.
Hoch
Merge-Konflikte mit main
PR ist 9 Tage alt (18.03.). Seitdem wurden 12+ Commits auf main gepusht (Feedback-System, Testimonials, Voting). Massive Konflikte in vercel.json, package.json, und möglicherweise lib/* Dateien erwartet.
Mittel
Vercel TS Runtime
Vercel Serverless Functions unterstützen .ts nativ, aber das Verhalten bei Edge Cases (dynamische Imports, ES Modules) kann abweichen. "npx tsc" lokal bestanden heißt nicht "Vercel baut korrekt".
Mittel
Ungetestete Runtime
21 Unit-Tests, aber kein Preview-Deployment getestet (Test-Plan zeigt "Deploy to Vercel preview" als offen). Typen stimmen, aber Runtime-Verhalten ist nicht validiert — z.B. Redis-Deserialisierung, JWT-Token-Handling, Stripe API Calls.
Mittel
JS/TS Koexistenz
Neue .js Dateien (unser Feedback-System, Radar) müssen neben .ts Dateien funktionieren. Biome-Config und tsconfig könnten unsere JS-Dateien blockieren oder mit Warnings überfluten.
Niedrig
Regressions in Types
Strikteste TS-Config + 3 Iterationen Review-Agenten. Typ-Sicherheit ist wahrscheinlich korrekt. Geringe Gefahr falscher Type-Casts (nur 1 "as" Cast im gesamten PR, dokumentiert).
05
Chancen

Was bringt es?

Langfristige Code-Qualität vs. kurzfristiges Risiko

Sofort
Type Safety
Compile-Time Fehler statt Runtime-Crashes. Jeder API-Endpoint hat klare Input/Output-Typen. IDE-Autocomplete und Refactoring werden dramatisch besser.
Sofort
Security Fixes
XSS in branded-error gefixt. Math.random() JTI durch crypto.randomUUID() ersetzt. Beides waren echte Vulnerabilities.
Kurzfristig
Dead Code Elimination
knip Detector findet ungenutzte Exports und Dateien. redis-typed.ts und oc.ts bereits entfernt. Weniger Code = weniger Angriffsfläche.
Mittelfristig
Konsistente Code-Qualität
Biome erzwingt einheitlichen Style, blockt unsichere Patterns (noExplicitAny, noDoubleEquals). Neue Beiträge werden automatisch geprüft.
Mittelfristig
Developer Experience
Type Definitions als lebende Dokumentation. Neue Entwickler verstehen Datenstrukturen sofort. Redis-Schema als Type statt als CLAUDE.md Kommentar.
Langfristig
Skalierbarkeit
Mit wachsendem Team (Contributors, KINN:PATE) wird Type Safety zum Sicherheitsnetz. Refactoring wird sicherer, API-Contracts sind explizit.
Netto-Effekt: -1.419 Zeilen trotz umfassender Type Definitions. Der Code wird nicht größer, sondern sicherer — ein seltenes Ergebnis bei TS-Migrationen.
06
Empfehlung

Nicht blind mergen.

Der PR ist gut — aber das Deployment-Risiko erfordert einen klaren Plan

Schritt 1
Preview Deploy
PR-Branch auf Vercel Preview deployen. Jeden Endpoint manuell testen: Auth, Fund, Gov, Admin. Der offene Testplan-Punkt "Deploy to Vercel preview" muss erledigt werden.
Schritt 2
Merge-Konflikte lösen
main in PR-Branch rebasen. Feedback-System (api/feedback/*.js, lib/feedback.js) muss als JS neben TS koexistieren. Biome/tsconfig darf unsere JS-Dateien nicht blockieren.
Schritt 3
Staged Merge
Merge in Low-Traffic Window (z.B. Sonntag). Sofort alle kritischen Flows testen: Signup, Fund, Admin-Login, Feedback-Versand. Rollback-Plan: git revert des Merge-Commits.
Bottom Line: Der PR bringt echten Mehrwert (Type Safety, Security Fixes, DX). Aber ein 100-Dateien Big-Bang Merge ohne Preview-Deployment wäre KINNlos riskant. Erst Preview testen, dann mergen.
